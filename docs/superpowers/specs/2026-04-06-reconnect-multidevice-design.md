# Reconnect & Multi-Device Support â€” Design Spec

**Date:** 2026-04-06
**Repository:** github.com/aneryu/kite

## Overview

Fix three usability issues with the current connection model:

1. **No reconnection** â€” Phone temporarily loses network â†’ must re-pair from scratch
2. **No daemon restart resilience** â€” Daemon restarts â†’ pairing code changes â†’ must re-pair
3. **No multi-device** â€” Only one browser can connect at a time; new browser replaces old

## Root Causes

The current architecture is a strict 1:1 pairing model with three layers of limitations:

**Signal Server (`signal/room.go`):**
- `Room` struct has exactly two slots: `Daemon Sender` and `Browser Sender` â€” no multi-browser support
- New browser replaces old one (`room.go:89-94`)
- Daemon disconnect deletes the room permanently (`room.go:155`)
- Room code cannot be re-registered (`room.go:62-63`)

**Kite Daemon (`src/main.zig`):**
- `global_rtc_peer: ?*RtcPeer` is a singleton â€” only one peer at a time (`main.zig:101`)
- New peer destroys old peer (`main.zig:324-327`)
- Pairing code generated fresh on every start (`main.zig:149`)
- No signal client reconnection logic

**Web Frontend (`web/src/`):**
- Clears localStorage on connection failure, forcing re-pair (`App.svelte:101,108`)
- Setup token is single-use (`auth.zig:37,43`)

## Design

### 1. Signal Server: Topic-Based Signaling

Replace the Room model with a Topic model. The signal server becomes a pure message router â€” no business logic, no role enforcement.

#### Data Model

```go
type Topic struct {
    Code       string
    Members    map[string]*Member  // key: member_id (server-generated UUID)
    LastActive time.Time
}

type Member struct {
    ID   string   // unique ID generated by server
    Role string   // "daemon" or "browser" â€” metadata only, not enforced
    Conn Sender
}
```

#### Protocol

**Client â†’ Server:**

| Message | Fields | Description |
|---------|--------|-------------|
| `join` | `pairing_code`, `role` | Join a topic |
| `relay` | `to` (member_id), `payload` | Send to a specific member |
| `broadcast` | `payload` | Send to all other members in the topic |

**Server â†’ Client:**

| Message | Fields | Description |
|---------|--------|-------------|
| `joined` | `member_id`, `members` (list of `{id, role}`) | Confirmation with current member list |
| `member_joined` | `member_id`, `role` | New member notification |
| `member_left` | `member_id` | Member departure notification |
| `relay` | `from` (member_id), `payload` | Forwarded message |

#### Topic Lifecycle

1. First member joins â†’ topic created
2. Member disconnects â†’ removed from topic, others notified via `member_left`
3. Last member leaves â†’ start 5-minute TTL
4. New member joins within TTL â†’ topic revived
5. TTL expires â†’ topic deleted

#### Key Principle

Signal server treats `role` as opaque metadata. It does NOT:
- Limit how many daemons or browsers can be in a topic
- Make routing decisions based on role
- Enforce any pairing logic

### 2. Kite Daemon Changes

#### Pairing Code & Setup Secret Persistence

- On first start: generate `pairing_code` (6 chars) and `setup_secret` (32 bytes, hex-encoded), save to `~/.config/kite/config`
- On subsequent starts: read from config file
- `kite setup --reset`: regenerate both and save

The config file already exists (stores `signal_url`). Add `pairing_code` and `setup_secret` fields.

#### Default Signal URL

Change default `signal_url` from `ws://localhost:8080` to `wss://relay.fun.dev`.

#### Multi-Peer Management

Replace `global_rtc_peer: ?*RtcPeer` with:

```zig
peers: StringHashMap(*RtcPeer)  // key: member_id from signal server
```

Behavior:
- `member_joined` (role=browser) â†’ create new `RtcPeer`, insert into map
- `member_left` â†’ destroy corresponding peer, remove from map
- PTY output â†’ iterate map, broadcast to all peers
- Input from any peer â†’ write directly to PTY (first-come-first-served)
- `relay` from member_id â†’ route to corresponding peer for SDP/ICE handling

#### Signal Client Auto-Reconnect

Current: `readLoop` exits silently on error, no retry.

Change to:
- Disconnect detected â†’ wait 2 seconds â†’ reconnect
- Exponential backoff: 2s, 4s, 8s, 16s, cap at 30s
- On reconnect: re-send `join` with same pairing code and role=daemon
- On `joined` response: create peers for any browsers already in the topic
- Reset backoff on successful connection

#### Protocol Adaptation

`handleSignalMessage` updated for new message types:
- `joined` â†’ store own `member_id`, create peers for each existing browser member
- `member_joined` (role=browser) â†’ create new peer, initiate WebRTC handshake
- `member_left` â†’ destroy corresponding peer
- `relay` â†’ route `payload` to peer identified by `from` field

#### `kite status` with QR Code

Extend `kite status`:
- Read `pairing_code`, `setup_secret`, `signal_url` from config
- Construct pairing URL: `https://relay.fun.dev/#/pair/{pairing_code}:{setup_secret}`
  (convert `wss://` to `https://` for the browser URL)
- Render QR code in terminal using pure Zig implementation
- Show status info (daemon running/not running, signal URL, pairing code)

### 3. QR Code Module (`src/qr.zig`)

Pure Zig QR code encoder + terminal renderer.

**Scope constraints (YAGNI):**
- Support Version 3-4 only (sufficient for ~80 char URLs)
- Error correction level: Low (maximum data capacity)
- Byte mode encoding only
- Terminal output using Unicode block characters: `â–ˆ`, `â–„`, `â–€`, ` ` (two pixel rows per character row)

**Estimated size:** ~400-500 lines.

**Interface:**

```zig
pub fn encode(allocator: Allocator, data: []const u8) !QrCode
pub fn renderTerminal(writer: anytype, qr: QrCode) !void
```

### 4. Web Frontend Changes

#### Auto-Reconnect with Full WebRTC Rebuild

Current `scheduleReconnect` only reconnects the signal WebSocket. Extend to:

1. Signal WebSocket drops â†’ 2s reconnect (existing)
2. Reconnect succeeds â†’ re-send `join` with stored pairing code and role=browser
3. `joined` response contains daemon in members list â†’ initiate WebRTC handshake
4. `joined` response has no daemon â†’ show "Waiting for daemon..." UI state
5. Later `member_joined` (role=daemon) â†’ initiate WebRTC handshake
6. DataChannel opens â†’ authenticate with session token from localStorage

#### Auth Flow Update

Current: `setup_token` is one-time, embedded in URL, consumed on first use.

Change to: `setup_secret` is persistent, embedded in URL, reusable.
- First visit with `#/pair/{code}:{secret}` â†’ exchange secret for session token, store token in localStorage
- Subsequent visits / reconnects â†’ use stored session token
- Session token expired (24h) â†’ re-exchange setup secret for new session token
- Setup secret rejected (after `kite setup --reset`) â†’ clear localStorage, show "Re-pair required" message

#### Don't Clear localStorage on Transient Failures

Current: connection timeout â†’ clear stored pairing code and session token.

Change to:
- Connection timeout / network error â†’ keep localStorage, show reconnecting UI
- Only clear localStorage on explicit `auth_rejected` from daemon (setup secret has been reset)

#### "Waiting" States

- Signal connected, no daemon â†’ "Waiting for daemon..."
- Signal disconnected â†’ "Reconnecting..." with spinner
- WebRTC negotiating â†’ "Connecting..."
- DataChannel open, authenticating â†’ "Authenticating..."

### 5. Auth Module Changes (`src/auth.zig`)

- Remove single-use guard on setup token (`setup_token_used` flag)
- `setup_secret` loaded from config file (not generated at runtime)
- `validateSetupSecret(secret)` â†’ returns session token (can be called multiple times)
- `validateSessionToken(token)` â†’ unchanged
- Session token TTL remains 24 hours
- Remove `renderQrCode` from auth.zig (moved to `kite status` using qr.zig)

## Behavior Summary

| Scenario | Before | After |
|----------|--------|-------|
| Phone temporary disconnect | Must re-pair | Auto-reconnect |
| Daemon restart | Must re-pair | Auto-reconnect (same pairing code) |
| Multiple devices | New device kicks old | All devices coexist |
| QR code lost | Scroll terminal history | `kite status` anytime |
| First-time pairing | Scan QR (one step) | Scan QR (one step, unchanged) |
| Default signal server | localhost:8080 | wss://relay.fun.dev |

## Out of Scope

- Device management UI (list/kick connected devices)
- End-to-end encryption beyond WebRTC's built-in DTLS
- Persistent session tokens across daemon restarts (session tokens are still runtime-only, 24h TTL)
- Rate limiting on signal server (existing rate limiting preserved)
